# OpenCS — 项目级开发约定

## OSS-First 实现原则（强制）

**实现任何模块前，必须先在 GitHub 调研同领域开源项目。** 优先在成熟 OSS 基础上集成或薄封装，而不是从零自建。

### 操作流程

每次开始一个新模块（含 phase plan 中的 task）时：

1. **调研**：`WebSearch` / `WebFetch` GitHub，找该领域 1–3 个候选库
2. **评估**：对每个候选用以下硬性标准过滤
3. **决策**：在该 phase 的 plan / commit message 里写明
   - `复用：<库名> @ <版本> — <理由>` 或
   - `自建：<理由 + 排除的候选>`
4. **优先复用 §6.6（spec 锁定清单）里已列的库**——这些已经做过决策，不要重新选

### 否决一个 OSS 候选的硬性条件（满足任一即可自建）

- 最近一次 commit 距今 > 18 个月（停更）
- 未发布稳定版（无 tagged release，或仅 0.x 早期版本）
- 核心 API 与 OpenCS 抽象冲突，封装成本 > 自建成本（要在决策记录里量化估计）
- License 不兼容（GPL/AGPL 之于商业自托管场景需评估）
- 安全：近 12 个月有未修复的 CVE，或缺少基本的输入校验/沙箱
- 体量超载：依赖树拉进 > 50MB 或引入与本项目无关的重模块（如某 utility 库强依赖整个深度学习框架）

### 不构成否决的理由（不要用这些当借口自建）

- "我可以写得更简洁"
- "我们的需求只用得上 20%"——薄封装即可，不要重写
- "Star 数不够"——star 是参考不是标准
- "中文社区没人用"——技术能力优先于本地生态

### 自建时仍要做的事

- 在 plan 里列出"自建模块的 OSS 等价物"，便于未来切换
- 接口设计参照已调研 OSS 的抽象（避免造异形轮子）

### 已锁定的依赖清单
见 `docs/superpowers/specs/2026-04-30-open-cs-mvp-architecture.md` §6.6（A: 直接复用 / B: 借抽象 / C: 必须自建）。

### License 兼容性策略

OpenCS 是企业自托管 + 开源项目，license 选择遵循：

| License | 直接使用 | 备注 |
|---|---|---|
| MIT / Apache-2.0 / BSD-2/3 / ISC | ✅ 直接用 | 主流宽松 license，无附加义务 |
| MPL-2.0 | ✅ 可用 | 文件级 copyleft，封装在独立文件即可 |
| LGPL-2.1/3.0 | ⚠️ 仅作为外部依赖动态链接，不修改源码 | 修改源需开源修改部分 |
| GPL-2.0/3.0 | ❌ 默认禁用 | 会传染整个 OpenCS；除非作为独立进程通信（如外部 CLI 工具）且经评审 |
| AGPL-3.0 | ❌ 默认禁用 | 网络服务也触发传染；商业自托管客户可能拒绝采用 |
| BSL / SSPL / Commons Clause | ❌ 默认禁用 | 非 OSI 认证开源，限制商业部署 |
| 自定义 / 不明 license | ❌ 不用 | 不可承担法律风险 |

**评审记录要求**：选用 LGPL 或例外引入 GPL 工具时，在 `docs/license-decisions.md` 留一条记录（库名、版本、用途、为何不能替换、隔离方式）。

## 模块研究产出物

### RESEARCH.md 格式

每个新模块（每个 phase 内的独立 task / 每个 `src/<module>/` 目录）实现前，必须在该模块根目录产出一份 `RESEARCH.md`：

```markdown
# <模块名> · OSS 调研

## 目标
<一句话：这个模块要解决什么问题>

## 候选对比

| 候选 | License | 最近 commit | Stars | 体量 | 评分 | 备注 |
|---|---|---|---|---|---|---|
| <repo URL> | MIT | 2026-03 | 8.2k | 1.4MB | ✅ | <一句话为什么合适/不合适> |
| <repo URL> | GPL-3.0 | 2024-11 | 3.1k | — | ❌ | License 否决 |
| <repo URL> | Apache-2.0 | 2022-08 | 12k | — | ❌ | 18 个月停更否决 |

## 决策
- **复用**：`<库名> @ <版本>`，封装在 `src/<module>/_<lib>_adapter.py`
- 或 **自建**：理由 = <逐个排除候选 + 量化封装成本>；OSS 等价物保留为 `<库名>` 以备 v1 切换

## 集成边界
<这个库暴露给项目其他部分的接口；哪些是 OpenCS 自己的抽象、哪些是直接透传>

## 升级与停更预案
<如何监控该库；停更时的迁移路径>
```

**强制规则**：没有 `RESEARCH.md` 不允许写该模块的实现代码。这条不是建议——是 phase 验收门禁。
